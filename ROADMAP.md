# Product Roadmap: IPInfo Enhancements (V2 Vision)

**Document Owner:** Product Management  
**Status:** Proposed / Backlog  
**Target:** Enterprise Scalability & Advanced Security Features  

---

## Executive Summary

The current iteration of the IPInfo microservice is highly robust, utilizing in-memory MaxMind databases, a zero-vulnerability Distroless Docker architecture, and a self-healing sidecar cron mechanism. It successfully fulfills our baseline requirements for high-concurrency IP geolocation and device fingerprinting. 

However, as we prepare the service to handle massive enterprise-level traffic patterns and function as a standalone, zero-trust infrastructure component, we must anticipate advanced security and performance requirements. This document outlines four strategic initiatives to be added to our engineering backlog. These features will elevate the service from a high-performing API to an industry-grade, mission-critical product.

---

## Proposed Initiatives

### 1. Threat Intelligence & Proxy Detection (Security Layer)
Knowing a user's location is insufficient if that location is being intentionally spoofed by malicious actors or botnets.

* **Implementation:** Integrate the `GeoLite2-Anonymous-IP` (or an equivalent commercial threat database) into our existing automated sidecar update loop.
* **Deliverable:** Append boolean threat flags to the JSON response (e.g., `"is_vpn": true`, `"is_tor": false`, `"is_datacenter": true`).
* **Business Value:** Allows consuming applications to automatically flag, challenge, or block suspicious traffic originating from anonymizing networks, dramatically reducing fraud and DDoS risks.

### 2. In-Memory LRU Caching (Performance Optimization)
While Go and Fiber are exceptionally fast, re-parsing the exact same User-Agent string or IP address during a high-traffic spike (or a localized DDoS attack) is an inefficient use of CPU cycles.

* **Implementation:** Implement a lightweight, thread-safe Least Recently Used (LRU) cache at the application level.
* **Deliverable:** Cache the JSON response for frequently seen IPs and User-Agents for a short TTL (e.g., 5 minutes). Requests hitting the cache will bypass the database lookup entirely.
* **Business Value:** Drastically reduces CPU utilization and maximizes throughput during sudden traffic surges without requiring additional server resources.

### 3. Graceful Shutdown Protocol (Reliability & UX)
Currently, automated container updates or restarts instantly terminate the process via a `SIGTERM` signal. This abruptly drops any HTTP requests that are actively being processed.

* **Implementation:** Implement OS signal trapping (`os.Interrupt`, `syscall.SIGTERM`) within the `main.go` entry point. 
* **Deliverable:** Upon receiving a shutdown signal, the server will stop accepting new connections and wait a configured duration (e.g., 5-10 seconds) for active requests to resolve before fully exiting.
* **Business Value:** Ensures true zero-downtime deployments and prevents degraded user experiences, data loss, or timeout errors during routine infrastructure updates.

### 4. Correlation IDs / Distributed Tracing (Observability)
As this microservice scales and gets consumed by various external gateways and platforms, debugging a failed request becomes highly complex without a unified tracing mechanism.

* **Implementation:** Enable Fiber's native `RequestID` middleware to inject a unique UUID (e.g., `X-Request-ID`) into every incoming request header.
* **Deliverable:** Every log entry generated by the application will include this Correlation ID, and the ID will be returned in the HTTP response headers.
* **Business Value:** Radically decreases incident resolution time (MTTR) by allowing engineers to trace a single request's journey across external API gateways, load balancers, and the IPInfo service.
